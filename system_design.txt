### D.1 LOGIN SEQUENCE

```mermaid
sequenceDiagram
    participant U as User
    participant FE as Frontend
    participant API as /api/auth/login
    participant AUTH as auth.controller
    participant SVC as auth.service
    participant DB as users table
    participant REDIS as Redis Cache
    participant OTP as otp.service
    participant EMAIL as SMTP

    U->>FE: Enter username/password
    FE->>API: POST /api/auth/login
    API->>AUTH: login({username, password})

    AUTH->>SVC: Check account locked?
    SVC->>DB: Query login_attempts
    DB-->>SVC: Return attempts count

    SVC->>REDIS: GET user:username
    REDIS-->>SVC: Cache miss

    SVC->>DB: SELECT * FROM users
    DB-->>SVC: Return user record

    SVC->>SVC: bcrypt.compare(password)
    SVC->>REDIS: SETEX user:username 300s

    alt 2FA Enabled
        SVC->>OTP: generateOTP()
        OTP->>REDIS: Store OTP
        OTP->>EMAIL: sendOTPEmail()
        EMAIL-->>U: OTP delivered
        API-->>FE: {requires_2fa: true}

        U->>FE: Enter OTP
        FE->>API: POST /api/auth/verify-otp
        API->>OTP: verifyOTP(user_id, otp)
        OTP->>REDIS: Validate OTP
        REDIS-->>OTP: OTP valid

        OTP->>DB: SELECT * FROM users
        DB-->>OTP: User record

        OTP->>JWT: sign(payload, secret)
        JWT-->>OTP: JWT token
        OTP->>REDIS: SETEX session:id 86400s
        API-->>FE: {token, user}
        FE->>FE: Store token in localStorage
    else 2FA Disabled
        SVC->>JWT: sign(payload, secret)
        JWT-->>SVC: JWT token
        SVC->>REDIS: SETEX session:id 86400s
        API-->>FE: {token, user, requires_2fa: false}
        FE->>FE: Store token in localStorage
    end

    FE->>FE: Redirect to /dashboard
```

### D.2 ATTENDANCE CHECK-IN/CHECK-OUT SEQUENCE

```mermaid
sequenceDiagram
    participant D as ESP32 Device
    participant API as /api/device/logs
    participant CTRL as device.controller
    participant SVC as attendance.service
    participant DB as attendance table
    participant RULES as attendance_rules table
    participant NOTIFY as notification.service
    participant SOCKET as Socket.IO

    D->>API: POST /api/device/logs
    API->>CTRL: receiveLogs({rfid, timestamp})
    CTRL->>DB: SELECT employee_id FROM employees
    DB-->>CTRL: Return employee_id

    CTRL->>SVC: createAttendance(employee_id, timestamp)
    SVC->>DB: SELECT today record
    DB-->>SVC: Return existing record or null

    SVC->>RULES: SELECT rules
    RULES-->>SVC: Return late_threshold, grace_period

    alt No existing record today
        SVC->>SVC: Check late status
        SVC->>DB: INSERT attendance (check_in)
        DB-->>SVC: Return new record
        SVC-->>CTRL: {status: CHECKED_IN}
    else Has check-in, no check-out
        SVC->>SVC: Validate duplicate scan
        SVC->>DB: UPDATE attendance SET check_out
        DB-->>SVC: Return updated record
        SVC-->>CTRL: {status: CHECKED_OUT}
    else Already completed
        SVC-->>CTRL: {message: Already completed}
    end

    CTRL->>NOTIFY: Create notification
    NOTIFY->>DB: INSERT notifications
    NOTIFY->>SOCKET: emit('attendance-update')
    SOCKET-->>FE: Real-time update
    API-->>D: {success: true}
```

### D.4 LEAVE REQUEST APPROVAL (CRITICAL FLOW)

```mermaid
sequenceDiagram
    participant E as Employee
    participant FE as LeavePage
    participant API as /api/leaves
    participant CTRL as leave.controller
    participant SVC as leave.service
    participant CREDIT as leaveCredit.service
    participant NOTIFY as notification.service
    participant DB as PostgreSQL
    participant A as Leave Approver
    participant AFE as AdminLeavePage

    E->>FE: Submit leave request
    FE->>API: POST /api/leaves
    API->>CTRL: createLeave(data)

    CTRL->>SVC: createLeave()
    SVC->>CREDIT: checkBalance(employee_id, type)
    CREDIT->>DB: SELECT leave_credits
    DB-->>CREDIT: Return balance

    alt Insufficient balance
        CREDIT-->>SVC: Balance error
        SVC-->>CTRL: Throw error
        API-->>FE: 400 Bad Request
    else Sufficient balance
        CREDIT-->>SVC: Balance OK

        SVC->>DB: SELECT employee_approvers
        DB-->>SVC: Return approver_id

        SVC->>DB: INSERT leaves (PENDING)
        DB-->>SVC: Return leave_id

        SVC->>NOTIFY: Notify approver
        NOTIFY->>DB: INSERT notifications
        NOTIFY->>SOCKET: emit('leave-request')

        SVC-->>CTRL: Success
        API-->>FE: {leave_id}

        NOTIFY->>A: Real-time notification
        A->>AFE: View pending requests
        AFE->>API: GET /api/leaves
        API->>CTRL: getLeaves()
        CTRL->>DB: SELECT leaves WHERE status=PENDING
        DB-->>CTRL: Return requests
        API-->>AFE: Leave list

        A->>AFE: Approve/Reject
        AFE->>API: PUT /api/leaves/:id/status
        API->>CTRL: updateStatus(id, action)
        CTRL->>SVC: updateStatus()

        alt APPROVED
            SVC->>DB: UPDATE leaves SET status=APPROVED
            SVC->>CREDIT: deductLeave()
            CREDIT->>DB: UPDATE leave_credits
            SVC->>DB: INSERT approval_logs
        else REJECTED
            SVC->>DB: UPDATE leaves SET status=REJECTED
            SVC->>DB: INSERT approval_logs
        end

        SVC->>NOTIFY: Notify employee
        NOTIFY->>DB: UPDATE notifications
        API-->>AFE: Success
    end
```

## H. KEY BUSINESS RULES
================================================================================

ATTENDANCE RULES
--------------------------------------------------------------------------------
1. Duplicate scan prevention: 2-minute window between scans
2. Late detection: timestamp > shift_start + late_threshold
3. Grace period: late_minutes < grace_period → not counted as late
4. Work fraction: partial days recorded as 0.5 or 1.0
5. Half-day types: MORNING or AFTERNOON

LEAVE RULES
--------------------------------------------------------------------------------
1. Balance check: request days <= available credits
2. Approval routing: employee_approvers table determines approver
3. Credit deduction: only on APPROVED status
4. Leave types: SICK, VACATION, MATERNITY, EMERGENCY, NO_PAY
5. Year-end conversion: unused credits → monetary (if convertible)

PAYROLL RULES
--------------------------------------------------------------------------------
1. Cutoff periods: defined in payroll_settings (1st & 2nd cutoff)
2. Pay day: defined per company
3. Components:
   - Basic salary: prorated by days worked
   - Overtime: approved + is_paid = true
   - Deductions: active employee_deductions
   - Late deductions: based on attendance_rules
   - Absent deductions: unworked days
4. Pay rules: multipliers for holidays/rest days
5. Status flow: PENDING → PAID (irreversible)

OVERTIME RULES
--------------------------------------------------------------------------------
1. Submission: before or after work date
2. Approval: required before inclusion in payroll
3. Paid flag: is_paid determines payroll inclusion
4. paid_in_payroll_id: links to specific payroll record

MAN HOUR REPORTING
--------------------------------------------------------------------------------
1. Required for: all worked dates (based on attendance)
2. Missing detection: attendance MINUS man_hour_reports
3. Approval: required for validation
4. Details: time_from, time_to, activity per entry

## J. EXTERNAL INTEGRATION POINTS
================================================================================

ESP32/RFID DEVICES
--------------------------------------------------------------------------------
Endpoint: POST /api/device/logs
Payload: { rfid_tag: string, timestamp: ISO8601, device_id?: number }
Response: { success: boolean, message: string, attendance?: object }
Flow:
  1. Device sends RFID scan
  2. Backend resolves employee_code → employee_id
  3. Attendance logic: check-in or check-out
  4. Response confirms action

SMTP EMAIL SERVER
--------------------------------------------------------------------------------
Configuration: smtp_settings table
Templates: email_templates table
Usage:
  - OTP delivery (auth.service)
  - Payslip delivery (queue.service → worker.js)
  - Notifications (emailWrapper.service)
Flow:
  1. Job added to Bull queue
  2. Worker picks up job
  3. PDF generated
  4. Email sent via nodemailer
  5. email_logs table updated

SOCKET.IO REAL-TIME
--------------------------------------------------------------------------------
Events emitted:
  - 'attendance-update': New attendance scan
  - 'leave-request': New leave for approval
  - 'overtime-request': New OT for approval
  - 'notification': New notification
Connection:
  - Client joins room: `user_${userId}`
  - Server emits to specific user room
